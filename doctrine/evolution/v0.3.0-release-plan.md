# v0.3.0 Release Plan: Precision, Applicability, And AI-Native Delivery

Status: Proposed implementation plan

Target: `v0.3.0`

Change class: Normative replacement, normative tightening, additive guidance, navigation, and editorial correction

Compatibility: Pre-1.0 minor; pinned consumers must review the migration before adoption

This is a non-normative planning record. It converts two substantive external reviews into decisions, owned tasks, acceptance criteria, and release evidence. Completing tasks is not itself proof that the release is safe: the release gate in T15 remains authoritative.

## 1. Release Thesis

`v0.3.0` should make the library smaller at the point of use and more precise at the point of obligation. It should preserve the depth of the reference corpus while giving adopters:

1. a compact core they can remember;
2. explicit normative strength and applicability;
3. a portable, expiring exception mechanism;
4. controls that can be measured, simplified, and retired; and
5. an AI-native change model that preserves this kernel:

> Intent is not implementation. Implementation is not evidence. Evidence is not authority. Authority is not enactment. Enactment is not outcome.

For AI-assisted delivery, objectives and outcome measures remain valuable for strategic and product interventions, but they are not mandatory fictions for every vulnerability fix, compatibility change, refactor, dependency update, or standing obligation.

| Five record families | **Take** | Evidence, authority, enactment receipts, and observations remain separately addressable within the families. |

| Independent verification by actor or model count | **Replace** | Require evidence diversity based on distinct failure modes and materiality. First-party deterministic tests are valid evidence but may be insufficient. |

| Multi-agent coordination invariants | **Take** | Add ownership, authority attenuation, bounded delegation, typed handoffs, cancellation, resume revalidation, and shared-state reconciliation. |
| Objective-to-KPI-to-task compilation for all changes | **Reject** | Every change needs a mandate; only strategic/product interventions require the full objective-to-measure-to-intervention-to-outcome overlay. |

## 3. Target Operating Shape

### 3.1 Core Doctrine

The compact entry point should express approximately ten memorable propositions without duplicating detailed controls:

1. optimise for stakeholder and operational outcomes, not activity volume;
2. make ownership, authority, and accountability explicit;
3. keep changes reviewable, reversible where feasible, and observable;
4. define and verify contracts at material boundaries;
5. automate repeatable evidence on the controlled change path;
6. design retried boundaries, failure handling, recovery, and reconciliation explicitly;
7. protect data, identities, dependencies, and delivery systems proportionately to risk;
8. preserve operability through telemetry, runbooks, rollback, and incident learning;
9. prefer simple, replaceable designs with controlled coupling; and
10. treat practices, standards, and tools as scoped defaults with explicit exceptions.

These are a drafting target, not text to copy without claim-level source review.

### 3.2 Applicability Overlay

The applicability pattern should support composition across at least:

- exposure;
- criticality and blast radius;
- data sensitivity and external obligations;
- operational model and recoverability;
- change autonomy;
- consumer compatibility; and
- observability of success and failure.

Profiles may name useful combinations such as a local utility, public service, critical production service, regulated data system, platform component, AI-assisted delivery path, or agent-enacted path. A profile activates controls; it does not create a new doctrine layer.

### 3.3 AI-Native Delivery Gates

The primary operational vocabulary should be:

1. **Admit mandate** — establish why work exists, ownership, scope, materiality, and justification class.
2. **Specify claims** — state what must be true and what evidence could discriminate it.
3. **Bound governed execution** — constrain model/agent authority when activation criteria are met.
4. **Produce candidate** — create an addressable change and disclose limitations.
5. **Challenge candidate** — gather evidence with sufficiently diverse failure modes.
6. **Authorise transition** — policy and accountable people decide for the exact candidate.
7. **Enact, observe, and close** — promote the authorised candidate, record receipts, evaluate runtime claims, and select the appropriate closure mode.

S0-S10 remains a detailed reference crosswalk, not the vocabulary every adopting team must encode as workflow states.

### 3.4 AI-Native Record Families

1. **Mandate** — justification class, owner, materiality, scope, non-goals, and optional strategic linkage.
2. **Governed execution** — authority envelope, inputs, allowed tools/data, outputs, limits, delegation, stop/escalation, and receipts.
3. **Candidate claim set** — immutable candidate identity, typed claims, limitations, affected surfaces, and required evidence.
4. **Challenge and decision family** — independently addressable evidence, findings, policy verdict, authority, approvals, and waivers.
5. **Enactment and observation family** — independently addressable deployment receipt, technical verdict, operational observation, rollback/containment, and optional outcome review link.

## 4. Release Gates

The release may be tagged only when all of the following are evidenced:

- ADRs 0028, 0029, and 0030 are accepted and cross-linked; ADR 0030 explicitly amends ADR 0024.
- New or materially changed normative claims have direct external grounding or a clearly labelled repository synthesis with source limits.
- `ENGINEERING.md` is a compact constitution and navigation surface, not a duplicate canonical corpus.
- Applicability and exceptions are portable and do not name an adopting organisation's tools, repositories, people, or live decisions.
- The AI SDLC uses seven operational gates, five record families, a governed-execution activation rule, typed claims, federated systems of record, and three closure modes.
- Challenge and authorisation remain separate; a waiver never rewrites technical evidence.
- The AI-native Mermaid diagram renders as four equal vertical charts, uses dark text on coloured nodes, and connects the bottom of each chart to the top of the next.
- The reasoning-evidence, run-budget/schema, worker-output trust, and checkpoint-resume inconsistencies are resolved.
- Navigation, glossary, references, semantic routes, checklist profiles, sitemap, migration notes, and changelog agree.
- Doctrine preflight, contract validation, Markdown link checks, Mermaid rendering, portability scans, and Git integrity checks pass on the release candidate.

## 7. Measures For The Release Work

Measure whether the revision improves adoption and precision, not how much Markdown it produces:

- number of overbroad claims corrected, with zero known P0 formulations left unresolved;
- percentage of sampled normative claims with explicit strength, applicability, canonical owner, and evidence route;
- number of duplicated umbrella obligations removed without orphaning canonical guidance;
- task-scenario success for low-risk maintenance, strategic product work, high-risk agent work, and externally controlled systems;
- reviewer time to identify the applicable rule and exception path;
- checklist items that change by profile/materiality rather than applying universally;
- broken-link, navigation-drift, Mermaid-render, schema-validation, portability, and preflight failure counts; and
- consumer migration actions that are explicit and bounded by applicability class.

Do not use lines removed, files added, task count, or diagram node count as success measures.

## 8. Deferred Work After `v0.3.0`

- Corpus-wide machine-readable metadata, only after a retrieval/adoption evaluation shows it is needed.
- A versioned run-contract v2 for budgets, checkpoint/resume identity, workflow dependencies, streaming triggers, or cross-contract coordination, unless T10 proves a minimal versioned addition is required for internal consistency now.
- Automated claim-level doctrine linting beyond low-risk structural checks.
- Additional applicability profiles based on adopter evidence; the portable library should not invent estate-specific control matrices.
- Removal or directory moves that would require a post-1.0 major compatibility event.
